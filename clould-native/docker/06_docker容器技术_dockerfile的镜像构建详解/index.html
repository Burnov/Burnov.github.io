dockerfile构建镜像原理介绍

dockerfile的详细介绍地址

我们通过docker build命令可以使用Dockerfile文件构建自己的image镜像。在生产环境中，本公司的业务镜像才是公司的核心内容；因此，基于dockefile去构建公司自己内部的业务镜像尤其重要；

一般情况下，我们会通过dockerfile结合Linux内置shell脚本一起来构建镜像文件，具体写法看业务需求；然后运维人员需要实现对镜像进行打包、分发、部署等操作；

在镜像管理方面一般情况下公司会搭建自己的k8s管理平台，或者去云厂家购买k8s管理平台，但是自己搭建的居多，因为可以定制自己公司的功能需求，而厂家提供的k8s平台则有很多的限制，不够灵活；

镜像介绍

1、docker镜像基于union file system将多个目录合并挂载到一个目录给容器使用；

​ 根据镜像的分层结构，基于overlay2的原理，如果不同的容器使用了底层相同的父镜像，则通过hash值来判断其底层镜像是否相同。如果相同，能够公用，而不会创建新的底层镜像文件；

2、最底层的是bootfs，镜像没有内核，使用的是宿主机的bootfs；

​ 镜像不包含内核，镜像里面也是没有bootfs文件的，但是又rootfs的根文件系统；

3、一个镜像是有一层或者多层合并而成，每一层称为是一个layer；

​ 镜像原理叫做“分层构建，联合挂载”，基于这个原理，每一个操作都会被分配为一个“层”。而镜像就是基于这些“层”联合起来的一个文件内容；

4、镜像可以基于其他镜像进行重新构建，被引用的镜像被称为父镜像；

​ 一般我们自己的业务镜像可以通过下载官方镜像后修改得到我们自己需要的镜像文件；

5、一个镜像可以同时被创建为多个容器；

6、镜像是只读的，任何的更改都不会直接修改镜像；

​ 真正进行数据修改的时候是在最上层的可写层（Writeable Container）进行数据修改的，不会修改任何镜像相关的数据内容；

容器启动的引导过程

在引导容器启动的时候底层Kernel内核提供虚拟化资源隔离功能，容器没有内核，自然也没有bootfs。因此，容器上只需要挂载rootfs作为根目录即可；所以，容器的启动过程只用引导rootfs启动即可；

dockerfile与镜像关系

dockerfile -> image -> container1/container2…

基于dockerfile构建公司内部的携带业务的image镜像，让后通过镜像去启动容器；且一个镜像文件可以启动多个容器；

docker的app应用程序内部开发流程介绍

如上图所示：

1、开发人员写好业务代码，以及相应的app程序；

2、将业务代码挂载关联到dockerfile中来；一般dockerfile的编写需要由运维人员来完成，但是这样的话，就对运维人员对业务了解提出了一定的要求；

3、基于dockerfile构建自己的镜像文件；基础镜像一般来自于公司的内部仓库，然后基于这些基础镜像来配置公司的业务镜像；配置好的业务镜像，也可以保存到公司内部的仓库中来；

4、编写yml文件，以及服务的启动脚本文件；其实就是docker-compose定义的容器编排启动脚本；

5、运行docker-compose即可；这样所有指定的业务容器依次启动；

6、通过docker-compose编写完成以后，我们可以通过PC进行访问测试；此时我们就能够正常访问到我们的业务数据了；

7、推送版本，继续新的版本升级，这里我们通过灰度部署，可以继续对新的业务版本进行下一轮的开发部署了；在这个过程中会关联到本地的gitlab来进行代码的统一管理；

对于我们经常使用的基础镜像，一般为centos、debian、Ubuntu、alpine，基础镜像的选择主要取决于用户的熟悉程度，越是熟悉就可以选择这种底层镜像；基础镜像中不建议添加过多的调试工具，如果调试工具过多，则镜像大小又会增大，这样消耗的硬件资源就会增多；

业务升级：不到万不得已，不要升级业务；
