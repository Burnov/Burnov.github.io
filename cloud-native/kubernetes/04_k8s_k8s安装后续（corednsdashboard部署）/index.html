kubernetes模块详解

K8S组件介绍回顾

当有一个数据请求发送过来的时候，我们会通过API-server模块获取用户的数据请求。此时，K8S需要验证请求者的认证信息，确认权限。然后确认是否为有效请求，最后返回数据。这一些列的操作过程中，我们需要协调各个模块之间的工作；

判断为正常有权限请求后，我们需要通过API-server去找到scheduler调度模块通过raft算法，获取最有node节点创建pod，而创建的过程则需要通过controllers节点来控制完成，他们之间都会通过API-server模块来实现数据交互，在这个过程中所生成的数据均需要存储与etcd的分布式存储上；

node节点则通过kubelet模块获取API-server发送过来的pod创建请求，根据指定的需求去创建pod。同时，通过kube-proxy来控制网络模块，管理容器间的网络节点；而pod的创建则需要底层引擎的支持，在这里我们提供是docker，而docker本身则自己调用的就是containerd容器来完成的容器创建；

kubernetes API Server

  • 提供端口默认值6443，可以通过参数“–secure-port”来修改默认值
  • 默认IP地址为非本地（Non-Localhost）网络端口，通过启动参数“–bind-address”设置该值
  • 该端口用于接收客户端、dashboard等外部HTTPS请求
  • 用于基于Tocken文件或客户端证书及HTTP Base的证书
  • 用于基于策略的授权

  • API链接测试：早期的kube-apiserver都需要通过curl 127.0.0.1:6443的方式来实现访问；到那时在1.20、1.21版本以后，API的链接都需要通过tocken，使用https的方式来进行链接，这样才能保证数据的安全性；而真正需要通过curl –cacert来指定证书才能访问k8s，但是一般只有开发才会需要这样去访问；

kube-scheduler

  • kubernetes调度器是一个控制面进程，负责将Pods指派到节点上；
  • 通过调度算法调度Pod列表的每一个Pod从可用的Node列表中的选择一个最适合的Node，并将数据写入到etcd中
  • node节点上的kubelet通过API-server监听到kubernetes Scheduler产生的Pod绑定信息，然后获取对应的Pod清单，下载Image，并启动容器
  • 调度策略：
    • LeastRequestedPriority 优先从备选节点列表中选择资源消耗最小的节点（CPU+内存 消耗最小）
    • CalculateNodeLabelPriority 优先选择含有指定Label的节点
    • BalancedResourceAllocation 优先从备选节点列表中选择各项资源使用率最均衡的节点

整个调度过程中，scheduler先会过滤掉不满足条件的node节点，然后选择满足条件中最优的node节点进行资源分配，发呢配好资源后又及时将数据跟新到etcd数据中！！！

具体的调度过程如下图所示：

第一步、需要创建pod，寻找可以承载该资源的node节点。pod需要10核cpu，而现在节点中如果node3为16核，node2为12核，node1为6核；

第二步、过滤不满足条件的node节点。这里node1就会被过滤掉；

第三步、rank nodes；对node减去10核CPU后进行比较，发现node3减去10核后最优；

第四步、将node3的数据写入到etcd中，进入pod真正创建的过程；

kube-controller-manager

  • kube-controller-manager包括一些子控制器（副本控制器、节点控制器、命名空间控制器核服务账号控制器等），控制器作为集群内部的管理控制中心，负责集群内部的Node、Pod副本、服务端点（Endpoint）、命名空间（Namespace）、服务账号（Service Account）、资源限定（Resource Quota）的管理，当某个Node意外宕机的时候，Controller Manager会即使发现并执行自动化修复流程，确保集群中的pod副本始终处于预期的工作状态；
  • controller-manager控制器每间隔5秒钟检查一次节点的状态
  • 如果controller-manager控制器没有收到来自节点的心跳，则将该node节点标记为不可达
  • controller-manager将标记为无法访问之前等待40秒钟
  • 如果node节点被标记为无法访问五分钟还没有恢复，controller-manager会删除当前node节点所有的pod并在其他可用节点重建这些pod；

因此，node节点要是挂了，一般要等待五分钟以上pod才能才在别的node节点上面重建；

  • pod高可用机制：
    • node monitor period：节点监视周期，5s
    • node monitor grace period：节点监视器宽限，40s
    • pod eviction timeout：pod驱逐超时事件，5m

kube-proxy

  • kubernetes网络代理运行在node上，它反映了node上kubernetes API中定义的服务，并可以通过一组后端进行简单的TCP、UDP核SCTP流转发或者在一组后代理，其实就是kube-proxy通过在主机上维护网络规则并执行链接转发来实现kubernetes服务访问。
  • kube-proxy运行在每个节点上，监听API Server中服务对象的变化，再通过IPtables或者IPOVS规则来实现网络转发；
  • kube-proxy不同的版本呢可支持三种工作模式：
    • UserSpace：k8s v1.1之前使用，k8s 1.2及以后就已经淘汰了；【它需要来回考数据】
    • IPtable：k8s 1.1版本开始支持，1.2开始为默认；【适合于小型环境，大环境下iptables规则太多，不合适】
    • IPVS：k8s 1.9引入到1.11的正式版本，需要安装Ipvsadm、ipadm、ipset工具包核加载ip_vs内核模块；【现在基本上使用这个】

k8s的网络架构下，我们分为三个网络：1、node的物理网络；2、pod的内部虚拟网络；然后链接物理网络核pod的虚拟网络还需要一个service网路，如上图所示；

iptables的网络架构模型如下图所示：

ipvs的网络架构模型如下图所示：

ipvs的模型相对于iptables的模型效率会更高一些，使用ipvs模型需要在运行kube-proxy的节点上安装ipvsadm、ipset工具包核加载ip_vs内核模块，当kube-proxy以ipvs代理模式启动时候，kube-proxy将验证节点上是否安装了ipvs模块，如果未安装，则kube-proxy将回退到iptables代理模式；

使用Ipvs模式，kube-proxy会监视kubernetes service对象核endpoint，掉用宿主机内核netlink接口以相应的创建ipvs规则并定期与kubernetes service对象endpoint对象同步ipvs规则，以确保ipvs状态与期望一致，访问服务时，流量将被重新定向到其中一个后端pod，ipvs使用哈希表作为底层数结构并在内核空间中工作，这意味着ipvs可以更快的重定向流量，并且在同步代理规则时具有更好的性能。此外，ipvs为负载均衡算法提供了更多选择，例如：rr轮询、lc最小链接数、dh目标hash、sh源hash、sed最短期望延迟、nq不排队调度等。

kubelet

kubelet是运行在每个worker节点的代理组件，它会监视已经分配给节点的pod，具体功能如下：

  • 向master汇报node节点的状态信息
  • 接受指令并在pod中创建docker容器
  • 准备pod所需要的数据卷
  • 返回pod的运行状态
  • 在node节点执行容器的健康检查

我们同样可以在/etc/systemd/system/目录下查看kubelet的服务配置文件；

etcd

etcd是coreOS公司开发的目前是kubernetes默认的使用的key-value数据存储系统，用于保存kubernetes的所有集群数据，etcd支持分布式集群功能，生产环境使用时需要为etcd数据提供定期的备份机制！！这一点非常重要，因为所有的数据均在etcd中进行存储。

DNS模块

coredns模块介绍

DNS负责为整个集群提供DNS服务，从而实现服务之间的访问。

  • sky-dns
  • kube-dns:1.18
  • coredns

早期使用sky-dns核kube-dns:1.18版本，之后改为了coredns模块；我们可以在官方网站https://github.com/coredns/coredns下面来查看对应的信息；

在没有配置coredns服务之前，我们的容器是无法通过域名去访问其他公网资源的；

且，coredns服务为插件化的，它有点类似于nginx的插件服务的模式，可以通过添加模块的方式来启动服务。因此，它也可以加入普罗米修斯的监控模块，实现对外接监控的功能。
