docker容器的卷详解

docker的数据卷基本概念

如果正在运行中的容器产生了新的数据或者修改了现有的一个已经存在的文件内容，那么新产生的数据将会被复制到读写层进行持久化保存，这个读写层也就是容器的工作目录，此即“写实复制（COW）copy on write”机制；

如下图是将对根的数据写入到容器的可写成，但是把/data中的数据写入到一个另外的volume中用于数据持久化；

为了避免数据丢失，因此，业务数据通常需要挂载再外面持久存储空间上，如下图所示：

多个容器使用共享的网络存储，是数据持久化存储方式的常见解决方案：

前面所有的容器都会挂载一个共享存储空间，这样就能解决数据存储的问题；

k8s的架构实现持久化存储则更为麻烦，但是原理大致相同。它使用一个CSI接口来对接其他第三方存储解决方案，这样就不会显示数据的存储系统平台核设备了；

我们回到docker容器技术，我们通过docker inspect可以看到对应的容器存储方案，如下图所示：

  • LowerDir – image镜像层（镜像本身是只读的）
  • UpperDir – 容器的上层（读写层）
  • MergedDir – 容器的文件系统，使用Union FS（联合文件系统）将lowerdir和upperdir合并给容器使用；
  • workDir – 容器再宿主的工作目录

数据卷（data volume）基本概念

数据卷是宿主机上的目录或者是文件，可以被直接mount到容器当中使用，实际生产环境中，需要针对不同的类型的服务，不同类型的数据存储要求做相应的规划，最终保证服务的可扩展性，稳定性以及数据的安装性；

docker容器的网络详解

docker的基本网络架构

docker容器网络的基本架构如上图所示；无论是多少个容器，每个容器中都只有一个eth0网卡，且这个网卡的对端通过veth桥接在docker0的网桥上，如果是内部容器之间的互通，他们则通过二层MAC地址进行数据交互。但是如果一旦要实现三层互通，则需要通过eth0的网关进行路由数据转发；

因此，容器的宿主机一定要开启路由转发功能！！！这个非常重要；

docker网络类型

  • bridge – 桥接网络
  • host – 仅主机网络
  • null – 封闭网络

host模型

host模型，使用的参数为 –net=host；

启动容器的时候指定使用host模型的化，那么新创建的容器不会创建自己的虚拟网卡，而是直接使用宿主机的网卡和IP地址，因此在容器里面查看到的IP地址信息和宿主相同，访问容器的似乎后也是直接使用宿主机IP+容器的端口即可。不过容器的其他资源必须文件系统、系统进程等还是和宿主机保持隔离

此模型的网络性能最好，但是个容器之间的端口不能相同，适用于运行容器端口比较固定的业务；

host模式不支持端口映射，当指定端口映射的时候会提示如下警告信息，使用主机网络模式时，将丢弃已指定的都那口：

null模型

null模型则比较容易理解了，当容器需要独立运行而不于外界交互的时候则采用这种网络模型；在真正的生产业务中使用较少；

container模型

container模型使用参数 –net=container 名称或ID指定；和卷一样，可以使用server端相同的挂载卷；这里指定server端的网络模型，客户端只要关联了container端的网络模型，则我们可以采用相同的网络；

使用此模型创建的容器需要之地那个和一个已经存在的容器共享一个网络，而不是和宿主机共享网络，新建的容器不会创建自己的网卡也不会配置自己的IP，二十和一个已经存在的被指定的容器IP和端口范围，因此这个容器的端口不能和被指定的端口冲突，除了网络之外的文件系统、进程信息等仍然保持相互隔离，两个容器进程可以通过IO网卡及容器IP进行通信；

网络总结

  • 容器网络需要开启路由转发功能 – net.ipv4.ip_forward
  • 容器网络总共有四种模型
    • bridge – 桥接模型，nat转换
    • host – 主机模型，直接获取主机功能
    • null – 无网络
    • container – 容器cs端模型，c端复制s端网络
